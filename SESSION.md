# SWELL — SESSION PROTOCOL

> **AIエージェントが作業セッション開始時に必ず従うプロトコルです。**
> このファイルを読むことでクレジット消費を最小化し、効率的に作業できます。

---

## セッション開始時の必須手順

新しいセッションが始まったら、以下の順番でファイルを読んでください。コードを読む前にこれらを必ず確認することで、無駄なファイル読み込みを防ぎます。

**Step 1 — CONTEXT.md を読む**（アーキテクチャ・ファイル構造・制約の把握）

**Step 2 — LATEST.md を読む**（現在の実装状態・未解決事項の把握）

**Step 3 — PLAN.md を読む**（次にやるべきタスクの把握）

**Step 4 — todo.md を読む**（細かいタスク状態の確認）

これら4ファイルを読めば、コードを広範囲に読まなくても作業を開始できます。

---

## クレジット消費エコ化ルール

### ファイル読み込みの最小化

作業に直接関係するファイルのみを読みます。CONTEXT.md に記載されたアーキテクチャを信頼し、全ファイルを読み直すことはしません。変更するファイルのみを読みます。

### 1回の指示で完結させる

ユーザーから指示を受けたら、関連する全ての変更を1回のセッションで完結させます。「少し試してみる」「段階的に確認する」という進め方は避け、計画を立ててから一気に実装します。

### チェックポイントの節約

チェックポイントは重要なマイルストーンのみに保存します。細かいバグ修正は複数まとめて1回のチェックポイントにします。

### サーバー再起動の最小化

`webdev_restart_server` はキャッシュクリアが必要な場合のみ使用します。TypeScript エラーの確認は `npx tsc --noEmit` で行います。

### テストの効率化

テストは変更した機能に関連するものだけ実行します。全テストを毎回実行することは避けます。

---

## 実装時の禁止事項

以下は CONTEXT.md にも記載されていますが、特に重要なため再掲します。

`Pressable` に `className` を使用してはいけません（`onPress` が発火しなくなります）。テーマカラーをハードコードしてはいけません（`useThemeContext()` を使います）。`pagingEnabled` と `snapToInterval` を同時に使用してはいけません。LLM（`server/_core/llm.ts`）を呼び出してはいけません（コスト発生）。

---

## PLAN.md の参照方法

PLAN.md には人間が編集したロードマップが記載されています。作業開始時に PLAN.md を読み、`[ ]` のタスクを上から順に実装します。実装完了後は todo.md の該当項目を `[x]` に更新し、LATEST.md の「現在の実装状態」セクションを更新してからチェックポイントを保存します。

## よくある作業パターン

### バグ修正

1. CONTEXT.md でアーキテクチャを確認
2. 問題のあるファイルのみを読む
3. 修正を実装
4. `npx tsc --noEmit` でエラー確認
5. todo.md を更新
6. チェックポイント保存

### 新機能追加

1. PLAN.md で該当タスクを確認
2. CONTEXT.md でデータフローを確認
3. 影響するファイルのみを読む
4. 実装（サーバー側 → クライアント側の順）
5. LATEST.md を更新
6. チェックポイント保存

### データソース追加

1. `server/api/` に新しいクライアントファイルを作成
2. `server/api/data-router.ts` に fetch 関数を追加
3. `server/api/trpc-data-router.ts` の `getByCategory` に組み込む
4. `hooks/use-real-time-data.ts` でデータマッピングを確認
5. キャッシュ TTL を `cache-service.ts` に設定

## 自己学習・デグレード防止ルール（2026-03-26 追加）

### デグレード防止ルール1: 矢印ボタン修正時のチェック
**過去のデグレード:** VideoCard 修正時に scrollToCard 関数が上書きされた

**防止方法:**
- VideoCard 修正時に scrollToCard 関数も確認
- Web 環境では scrollToOffset ではなく scrollToIndex を使用
- 修正前後で矢印ボタン動作確認を必須

**チェックリスト:**
```
修正前: git diff app/(tabs)/index.tsx で scrollToCard 関数を確認
修正後: 矢印ボタン左右両方の動作確認（Web + ネイティブ）
```

---

### デグレード防止ルール2: YouTube サムネイル取得の失敗防止
**過去のデグレード:** YouTube API キーなしでサムネイル取得ができない

**防止方法:**
- YouTube API キーは不要（公開情報）
- サムネイル URL は `https://i.ytimg.com/vi/{VIDEO_ID}/hqdefault.jpg` で直接取得
- トレンド動画取得方法を事前に確認

**チェックリスト:**
```
実装前: YouTube トレンド動画取得方法を確認（API キー必須か不要か）
実装後: SOCIAL タブでサムネイル表示確認
```

---

### デグレード防止ルール3: WebView Web 互換性の確認
**過去のデグレード:** Web 環境で WebView が動作しない

**防止方法:**
- Web 環境では WebView ではなく iframe を使用
- Platform.OS === 'web' で分岐処理を必須
- ネイティブ環境と Web 環境で異なるコンポーネント使用

**チェックリスト:**
```
実装前: Platform 判定が必要か確認
実装後: Web + ネイティブ両方で動作確認
```

---

### デグレード防止ルール4: モックデータ削除の確認
**過去のデグレード:** モックデータが複数箇所に存在し、削除漏れがある

**防止方法:**
- モックデータ削除時は全ファイルを grep で確認
- 設定画面・テストファイル・データソースマネージャーなど複数箇所を確認
- 削除後は必ずテストで動作確認

**チェックリスト:**
```
削除前: grep -r "mock" で全検索
削除後: 各タブで実データ表示確認
```

---

### デグレード防止ルール5: API 制限の事前確認
**過去のデグレード:** Manus Data API ラッパーに制限があり、本番環境で使用不可

**防止方法:**
- 外部 API を使用する場合は直接 fetch を使用
- Manus ラッパーは開発環境のみ
- API 制限がないことを事前に確認

**チェックリスト:**
```
実装前: API 制限の有無を確認
実装後: 1000万ユーザー規模での API 呼び出し削減率を計算
```

## Session 保持ファイル運用ルール

### ファイル構成
- **SESSION.md** （このファイル）: 修正履歴・教訓・デグレード防止ルール
- **LATEST.md**: 現在の実装状態・未解決事項
- **PLAN.md**: 次にやるべきタスク
- **todo.md**: 細かいタスク状態

### 修正時の必須手順
1. SESSION.md に修正内容を記録
2. todo.md に完了タスクをマーク
3. LATEST.md を更新
4. git commit -m "修正内容" でコミット
5. webdev_save_checkpoint でチェックポイント保存

### デグレード検出時の対応
1. SESSION.md のデグレード防止ルールを確認
2. 過去の修正履歴から原因を特定
3. 修正を実装
4. SESSION.md に新しいデグレード防止ルールを追加
5. git commit + webdev_save_checkpoint
